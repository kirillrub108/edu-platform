# Edllm изнутри: как PPTX превращается в озвученный видеоурок — карта системы

> **Площадка:** vc.ru (точка входа в серию; читается и на Habr) · **Теги:** SaaS, EdTech, архитектура, AI, бэкенд · **Время чтения:** ~9 мин

**TL;DR.** Edllm — это LMS, у которой одна подписная фича: загружаешь PPTX и текст лекции, а на выходе получаешь готовый озвученный MP4-урок и раздаёшь его студентам. Внутри — FastAPI на async-ядре, Celery с четырьмя очередями, локальный AI-стек (LLM + vision + TTS) и биллинг по кредитам. Эта статья — карта: что из чего состоит и почему так, со ссылками на технические разборы каждого узла.

---

## С чего всё началось

Преподаватель тратит вечер не на лекцию, а на то, чтобы её *записать*: поставить свет, не сбиться, перемонтировать. Мы решили убрать этот вечер. Идея Edllm проста до неприличия: **PPTX + сценарий лекции → нарезка по слайдам → озвучка → видео**. Преподаватель пишет текст, как привык, а сборку берёт на себя конвейер.

Звучит как «склеить картинки с аудио». На практике за этим — пять инженерных задач, каждая из которых тянет на отдельную статью:

- как ужиться async-бэкенду и синхронному Celery в одном кодовом дереве;
- как разрулить очереди, чтобы письмо не висело за тяжёлым рендером видео;
- как не ждать всю озвучку перед началом кодирования;
- как сделать auth без единого токена в JavaScript;
- как гонять LLM и vision локально, а не платить облаку за каждый слайд.

Дальше — карта. Каждый узел разобран отдельно; ссылки в конце разделов ведут в технические статьи серии.

## Вид сверху

```text
            Браузер (Nuxt 3 SPA, авторизация на httpOnly-куках)
                         │   /api/v1/*
                         ▼
              FastAPI (async, asyncpg / AsyncSession)
        тонкие роутеры → толстые сервисы → PostgreSQL
                         │   .delay() / .apply_async()
                         ▼
                       Redis  ──►  очереди:  video · vision · quiz · celery_email
                         │
        ┌────────────────┼─────────────────┬──────────────────┐
        ▼                ▼                 ▼                  ▼
   celery_video     celery_vision     celery_quiz        celery_email
   (sync Session, psycopg2)            (+ beat: единственный планировщик)
        │
        ▼
   LLM (qwen3) · Vision (Qwen-VL) · Silero TTS · FFmpeg  →  storage (MP4) + DB
```

Принцип «тонкие роутеры / толстые сервисы / синхронные таски» проходит через весь бэкенд. Роутер только парсит, проверяет права и делает 1–3 вызова в `services/`. Вся логика — в сервисах. Всё, что долго (рендер слайдов, vision-разбор, генерация квиза, рассылка), уезжает в Celery.

Один путь данных целиком:

```text
Browser → /api/v1/* → FastAPI router → Pydantic → service / async SQLAlchemy → PostgreSQL
долгие задачи → Celery.delay() → Redis → воркер очереди → LLM/TTS/FFmpeg → storage + DB
прогресс → SSE-стрим (/lessons/{id}/progress-stream) с поллингом /task-status как фолбэком
```

## Узел 1. Два мира в одном репозитории: async-API и sync-Celery

FastAPI у нас полностью асинхронный: `asyncpg` и `AsyncSession` в каждом роутере и сервисе. А Celery работает в prefork-режиме, где асинхронный драйвер БД упирается в greenlet-ошибки. Решение — провести чёткую границу: **таски берут отдельный синхронный движок**, а URL для него выводится из единственной настройки `DATABASE_URL` простым `replace`:

```python
# в каждом Celery-таске — синхронная сессия, не AsyncSession
_sync_url = settings.DATABASE_URL.replace("+asyncpg", "+psycopg2")
sync_engine = create_engine(_sync_url, pool_pre_ping=True)
SyncSession = sessionmaker(bind=sync_engine, expire_on_commit=False)
```

Это не «недоделали async везде» — это сознательная архитектура с одним конфигом и двумя мирами. Там же живёт неочевидная ловушка с `onupdate=func.now()` и `eager_defaults`, без которой строка падает с `MissingGreenlet` прямо при сериализации ответа.

→ Технический разбор: **«Async-API и sync-Celery в одном проекте»** (статья 2).

## Узел 2. Четыре очереди, один планировщик

Видео-рендер, vision-разбор, генерация квизов и письма — это четыре очень разных нагрузки. Гонять их в одной очереди — значит позволить тяжёлому видео заблокировать отправку письма с подтверждением почты. Поэтому у нас **четыре очереди и по воркеру на каждую**, а планировщик (`beat`) встроен ровно в один из них — иначе ежедневная задача очистки запускалась бы дважды.

Поверх этого — приоритеты по тарифу пользователя. Тонкость: брокер у нас Redis, а на нём **меньшее число = больший приоритет** (0 раньше, 9 позже) — обратно RabbitMQ.

→ Технический разбор: **«4 очереди, один beat и приоритеты на Redis»** (статья 3).

## Узел 3. Стриминг озвучки в кодировщик

Сборка видео — это N слайдов, каждому нужна озвучка (TTS), потом кодирование (FFmpeg). Наивный путь — «озвучить всё, потом закодировать всё» — оставляет кодировщик простаивать всё время, пока идёт TTS. Мы соединили два пула потоков через `as_completed`: **как только готов WAV слайда k, он сразу уходит на кодирование**, перекрываясь с озвучкой следующих слайдов. Латентность сборки падает примерно вдвое.

→ Технический разбор: **«Стриминг TTS→FFmpeg через `as_completed`»** (статья 4).

## Узел 4. Auth, которого нет в JavaScript

Мы не используем `Authorization: Bearer` и не кладём токены в `localStorage`. Доступ — на **httpOnly-куках**, которые JS вообще не видит; защита от CSRF — **double-submit** (заголовок `X-CSRF-Token` должен совпасть с не-httpOnly кукой), а протухший access-токен чинится прозрачно: на 401 фронт делает **один** `refresh` на всю пачку запросов (singleflight) и повторяет исходный.

Отдельно — про AI-гейтинг: ни один эндпоинт, дёргающий LLM/vision/TTS, не уезжает в прод без проверки верифицированной почты. Это не «договорённость в ревью», а **тест, который роняет CI**.

→ Технический разбор: **«Auth без Bearer и localStorage + CI-гейтинг AI»** (статья 5).

## Узел 5. Локальный AI и экономика кредитов

Все наши AI-провайдеры говорят на одном протоколе (OpenAI-совместимом), поэтому **смена провайдера — это правка `.env`, а не кода**. Дефолт можно держать локальным: `qwen3` для текста, локальная `Qwen-VL` для разбора слайдов (vision вместо OCR), и **Silero TTS в контейнере** — стек без облачных счетов и с приватностью данных «у себя».

Расход считается не «квотами на запросы», а **кредитами**: длинные операции сначала *резервируют* кредиты, потом *списывают* или *возвращают* по факту. Это и про деньги, и про надёжность — баланс не уйдёт в минус и не спишется дважды на задаче, которая упала на середине.

→ Продуктово-технический разбор: **«Локальный AI вместо облака + биллинг по кредитам»** (статья 6).
